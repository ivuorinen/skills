Lesson 0001 · Nitpicker dispatch

From /nitpicker <text> to a running command

Your win for this lesson: by the end you can take any /nitpicker … line and say, with certainty, which single command file runs and why — including the two cases that trip people up (aliases, and unknown words). That is the foundation the whole "review and extend confidently" mission stands on.

The shape of an invocation

Everything after /nitpicker is parsed as one small grammar (SKILL.md → Dispatch): 

/nitpicker [command] [extra instructions]

There is no argument parser, no flags library. It is just text: the first word is a command candidate, and everything after it is free-text instructions that constrain the run (scope, focus, thresholds). Two of those instruction words are special — inline and changed-files — and may appear anywhere (_conventions.md → Modifiers).

The pipeline, step by step

  1. Split the text. Take the first whitespace-delimited word as the command candidate; keep the remainder as extra instructions.
  2. Resolve the first word against the command tables in SKILL.md. Both canonical names (tests) and aliases (test-auditor) resolve to the same file.
  3. Fall back on a miss. If the first word matches nothing — or there is no text at all — the router runs the default audit command and treats the whole text as extra instructions. A typo never errors; it audits.
  4. Load in order. Read commands/_conventions.md first (it binds every command — severity, findings store, rules), then read commands/<resolved>.md.
  5. Execute exactly one. Run that single command with the extra instructions applied. The router never chains commands on its own — if a command tells you to run another first, you follow that, but the router itself dispatches one and only one.

Three invocations, worked

You typeFirst word resolves toWhat runs
/nitpicker tests only src/ canonical tests commands/tests.md, scoped to src/
/nitpicker test-auditor alias → tests the same commands/tests.md
/nitpicker frobnicate the code no match → default commands/audit.md, with "frobnicate the code" as instructions

This is exactly why invoking teach through the published plugin — which does not know that command yet — would have fallen through to audit: an unknown first word is not an error, it is the default.

Why this matters for extending

Routing and registration are two sides of one coin. A command only resolves in step 2 because it has a row in a SKILL.md command table. Add a command and you must do both: create commands/<name>.md and add its table row. The validator validate-skill.py enforces this as a strict 1:1: a file with no row, or a row with no file, fails CI. When you review a command PR, that is the first thing to check.

Check yourself (recall, don't re-read)

Answer each out loud before opening it. Retrieval is what builds retention.

You type /nitpicker teardown everything and there is no teardown command or alias. What runs?

The default audit command, with "teardown everything" passed through as extra instructions. Unknown first word → fallback, never an error.

In what order are files read once a command resolves, and why that order?

_conventions.md first, then the command file. The conventions bind every command (severity, findings store, modifiers), so they must be in force before the command's own instructions are applied.

You add commands/leaks.md but forget the SKILL.md table row. Two things break — what are they?

(1) The word leaks resolves to nothing, so /nitpicker leaks falls through to audit. (2) validate-skill.py fails the 1:1 check in CI — a file with no row.
